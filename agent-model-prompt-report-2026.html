Agent 框架篇

OpenAI Agents SDK
2025年3月发布
采用 Instructions + System Prompt 双层设计。Instructions 是开发者定义的简短行为指令，框架自动将其注入 system prompt 并结合内置护栏（Guardrails）。强调 Tool Calling 为第一公民，Agent 通过函数签名自动理解工具语义。支持 Handoff 机制实现 Agent 间任务交接，交接时自动传递上下文摘要。
Instructions层 Tool Calling Handoff Guardrails
🧠
Anthropic Claude
Extended Thinking · 原生Tool Use
System Prompt 拥有最高优先级，Claude 被设计为对 system prompt 高度遵从。Extended Thinking 模式下，prompt 会触发模型内部深度推理（不可见），只在最终输出时展示思考过程。Tool Use 通过 JSON Schema 定义函数，prompt 自动注入工具描述和调用格式。支持 200K token 上下文，prompt 可包含大量示例和文档。
System Prompt优先 Extended Thinking 200K上下文 JSON Schema
🔗
LangChain / LangGraph
模板化 · 状态图驱动
Prompt 采用 ChatPromptTemplate 模板化设计，支持 System / Human / AI 多角色消息拼接。LangGraph 进一步引入 状态图（StateGraph），每个节点可拥有独立 prompt，节点间通过共享 state 传递上下文。支持 conditional edges 实现动态 prompt 路由，复杂 Agent 逻辑被分解为多个专业化 prompt 节点。
模板化 StateGraph 多节点Prompt 条件路由
👥
CrewAI
角色扮演 · 多Agent协作
核心是 Role-Based Prompt：每个 Agent 被赋予明确的角色（Role）、目标（Goal）和背景故事（Backstory），这三者组成该 Agent 的 system prompt。任务（Task）拥有独立的 description 和 expected_output，执行时框架将角色 prompt + 任务描述组合后发给 LLM。支持 顺序执行和层级执行两种协作模式。
Role+Goal+Backstory Task描述 顺序/层级执行
💬
AutoGen (Microsoft)
对话驱动 · 自然语言协调
Agent 间通过自然语言对话协调工作，而不是预定义的流程图。每个 Agent 的 system_message 定义其能力和行为边界。采用 ConversableAgent 基类，prompt 设计强调"你是什么角色、你能做什么、你应该如何与其他 Agent 交流"。支持代码执行反馈回路，Agent 可以写代码→执行→根据结果自我修正 prompt。
自然语言协调 system_message 代码执行回路
🏭
MetaGPT
SOP注入 · 工程化Prompt
将软件工程SOP（标准操作流程）编码到 prompt 中。每个 Agent（产品经理、架构师、工程师等）的 prompt 包含该角色的标准工作流程和产出物模板。引入 结构化文档（如PRD、设计文档）作为 Agent 间通信介质，而非纯自然语言对话。通过 约束编码确保产出物符合工程规范。
SOP编码 结构化文档 角色专业化 约束编码

Claude Opus 4 / Sonnet 4
Anthropic · 200K上下文
对 System Prompt 的遵从度业界最高，Claude 被训练为严格遵循 system 角色的指令。Extended Thinking 让模型在回答前进行不可见的深度推理。prompt 设计强调 清晰的角色定义 + 具体约束 + 正面指令（"Do X"而非"Don't do Y"）。Tool Use 的 function description 质量直接影响调用准确率。
最高遵从度 Extended Thinking 正面指令优先
🌌
Gemini 2.5 Pro
Google · 1M Token 上下文
1M token 超长上下文窗口，可将整本书、完整代码库或大量文档直接放入 prompt。支持 Thinking Mode（类似CoT），将推理过程作为响应的一部分返回。原生多模态，prompt 中图文音视频可任意混合。System Instruction 独立于用户消息，支持对模型行为的精细控制。
1M上下文 Thinking Mode 原生多模态
🔮
DeepSeek V3 / R1
深度求索 · 开源先锋
R1 的思维链（CoT）输出透明化是其最大特色——模型的推理过程对用户可见，prompt 可以引导或约束思考方向。采用 System + User 双角色体系，prompt 结构简洁高效。V3 作为通用模型，128K 上下文 + 极低价格使得长 prompt 策略（如Few-shot + 大量文档）非常经济可行。
CoT透明化 System+User 128K经济 开源生态
☁️
Qwen 3 (通义千问)
阿里 · 多语言多模式
支持 Thinking / Non-Thinking 双模式切换，通过 prompt 中的开关控制是否启用深度推理。对中文 prompt 理解天然优秀，多语言混合 prompt 处理流畅。工具调用（Function Calling）支持 并行多工具调用，prompt 中可同时描述多个工具并按需选择。开源版本支持本地 prompt 定制和微调。
双模式切换 中文优化 并行工具调用
🦙
Llama 4 (Meta)
开源 · MoE 架构
采用 MoE（混合专家）架构，Scout/Maverick 两个版本针对不同场景。作为开源模型，prompt 行为可通过微调完全定制，社区贡献了大量优化后的 prompt 模板。原生支持 128K 上下文和多语言。prompt 设计上倾向于简洁的 system prompt + 清晰的任务描述，过度复杂的 prompt 结构反而不利。
MoE架构 开源可定制 简洁prompt
核心发现：2026年主流模型的 System Prompt 遵从度差异显著——Claude ≈ GPT-4o > Gemini > DeepSeek ≈ Qwen > Llama。在 Agent 框架中选择模型时，System Prompt 遵从度是决定 Agent 行为可控性的关键因素。
03

Prompt 工程趋势

Chain-of-Thought / Tree-of-Thought

CoT 已成为标配——几乎所有现代模型都内置了思维链能力。ToT 进一步允许多分支探索，但主要在 Agent 框架层实现而非模型原生。提示词趋势：从"Let's think step by step"进化为指定思考框架。

ReAct (Reasoning + Acting)

推理与行动交替进行的范式成为 Agent 框架的默认设计模式。Prompt 需要同时描述"如何思考"和"何时行动"，Thought→Action→Observation 循环已标准化。

Structured Output & JSON Mode

从"希望输出JSON"到"必须输出符合该Schema的JSON"，结构化输出从愿望变成契约。OpenAI、Anthropic、Google 均原生支持 JSON Schema 约束，prompt 中不再需要写复杂的格式说明。

Few-shot 精简化

模型能力提升使得 Few-shot 示例数量趋于精简——2-3个高质量示例优于10个普通示例。关键是示例的多样性和边界覆盖，而非数量。

System Prompt 工程化

System Prompt 从"一段话"进化为结构化文档：角色定义→行为准则→工具说明→输出格式→边界约束。分层设计、版本管理、A/B测试成为标配。

MCP & Function Calling 融合

Model Context Protocol (MCP) 将工具/资源的 prompt 注入标准化。Agent 不再需要手动拼接工具描述，MCP Server 自动提供结构化的工具定义注入到 prompt 中。

上下文压缩与长Prompt优化

随着上下文窗口增大（1M token），prompt 的"信息密度"成为新焦点。关键信息前置（Primacy Effect）、重复强调（Recency Effect）、分段标记等技术成为长 prompt 设计的必修课。

一句话总结：2026年的 Prompt 已经从"提示"进化为精密的行为编程语言。Agent 框架将它模块化、工程化，模型将它内化为核心能力。最好的 prompt 不是最多的词，而是最精确的结构 + 约束 + 示例的组合。
